# 第 20 课：Worker 与自动调度

## 1. 本课定位

这一课把你从“用户点一下触发一次处理”的思维，带到“系统自己持续运行”的思维。Worker 是 `ResearchOS` 具备研究管线属性的关键模块。

## 2. 学完你应该能回答的问题

- 为什么单靠 API 进程不足以支撑完整科研系统。
- Worker 在 `ResearchOS` 中承担哪些后台职责。
- `APScheduler` 如何与 Topic 调度、日报、图谱维护、闲时处理结合。
- 为什么单实例锁和心跳文件是必要的。

## 3. 学习前准备

- 通读 `apps/worker/main.py`。
- 阅读 `packages/ai/ops/daily_runner.py`。
- 浏览 `packages/ai/ops/idle_processor.py`。

## 4. 详细讲解

### 4.1 Worker 解决的是“持续运行”问题

API 进程更适合处理用户即时请求，但研究系统还有很多事情不应该等用户来点：

- 定时抓取主题论文
- 生成每日简报
- 维护图谱
- 在空闲时补做低优先级任务

这些任务天然属于 Worker。

### 4.2 `apps/worker/main.py` 体现了完整调度思路

从这个文件里你能看到几个很成熟的工程点：

- `BlockingScheduler` 作为调度器
- 使用 CronTrigger 定义周期任务
- 单实例锁避免多个 worker 同时跑
- 心跳文件支持健康检查
- 重试与指数退避
- 闲时处理器独立运行

这说明 Worker 不是一个简单 while 循环脚本，而是正式后台进程。

### 4.3 Topic 调度为什么很关键

`topic_dispatch_job()` 把 Topic 对象和自动抓取联到了一起。它的价值在于：

- 把研究兴趣转化成长期订阅
- 让系统按规则自动跟踪
- 避免用户每天手动搜索

这是“科研助手”比“论文搜索器”更进一步的核心标志。

### 4.4 简报与图谱维护体现了后台系统思维

日报和每周图谱维护都不属于用户即时交互，却直接影响用户体验：

- 日报让用户固定频率消费结果
- 图谱维护让结构性洞察随时间更新

它们说明系统已经从“提供功能”变成“持续服务用户的研究节奏”。

### 4.5 为什么要有单实例锁与心跳

这两个细节特别值得新手学习：

- 单实例锁防止同一数据目录有多个 Worker 重复执行任务。
- 心跳文件让外部能知道 Worker 是否还活着。

这类设计看似不起眼，实际上是把脚本升级为长期运行服务的关键。

## 5. 参考代码对照

### 5.1 对照 `reference/claw-code-main`

`reference/claw-code-main` 更强调本地研究资产与工作台交互。对照它可以帮助你理解：当前 `ResearchOS` 之所以额外保留 Worker 进程，是因为它不仅要展示资产，还要持续运行后台研究管线。

## 7. 动手任务

1. 列出 Worker 当前负责的全部后台任务。
2. 解释每类任务分别为什么不适合放在 API 请求链路中执行。
3. 说明单实例锁、心跳、重试各自缓解什么风险。

## 8. 验收标准

- 你能解释 Worker 进程存在的必要性。
- 你能画出 Topic 调度和日报生成的高层流程。
- 你能说明后台工程细节对稳定性的价值。

## 9. 常见误区

- 误区一：把 Worker 看成“顺手加的定时脚本”。
- 误区二：忽视后台任务与用户体验之间的联系。
- 误区三：低估单实例锁和健康检查的重要性。
