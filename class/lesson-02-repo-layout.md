# 第 02 课：仓库结构入门

## 1. 本课定位

这一课解决的是“去哪里看代码”的问题。对大型仓库来说，目录结构就是第一层架构图。如果你不能准确区分哪些目录是主代码、哪些是运行数据、哪些是工程资产，你后面读任何模块都会迷路。

## 2. 学完你应该能回答的问题

- `apps/`、`packages/`、`frontend/` 各自承担什么职责。
- `data/`、`projects/`、`runtime/` 为什么不能和源码混看。
- `scripts/`、`tests/`、`infra/`、`reference/` 在学习时分别应该怎么用。
- 当前仓库和 `reference/claw-code-main` 的目录组织差异说明了什么。

## 3. 学习前准备

- 直接浏览仓库根目录、`apps/`、`packages/`、`frontend/src/`、`reference/claw-code-main/`。
- 用文件浏览器或终端先浏览仓库根目录。
- 不要求记住所有子目录，但需要先知道顶层目录有哪些。

## 4. 详细讲解

### 4.1 用“五层职责”理解根目录

从当前仓库根目录和主要子目录可以直接归纳出很清楚的分层方式：

- 产品源码：`apps/`、`packages/`、`frontend/`
- 运行数据：`data/`、`projects/`
- 工程资产：`scripts/`、`tests/`、`infra/`、`skills/`
- 参考资料：`reference/`
- 可再生产物：缓存、临时产物、打包中间物

这意味着你读代码时，最先应该盯住的是源码层；而你排查运行结果时，往往要去运行数据层和工程资产层。

### 4.2 `apps/` 和 `packages/` 的区别

很多 Python 项目都把所有逻辑塞在一个 `app/` 下，但这里做了更明确的区分：

- `apps/` 是应用入口层，负责把系统“跑起来”，例如 API、Worker、Desktop Server。
- `packages/` 是能力层和领域层，负责真正的业务逻辑、Agent runtime、集成能力、存储层。

一个简单记忆法：

- `apps/` 更像“程序从哪里进来”。
- `packages/` 更像“程序核心能力住在哪里”。

### 4.3 为什么 `frontend/` 单独存在

`frontend/` 独立出来，说明前端不是后端附属页面，而是完整应用。它有自己的：

- 依赖管理
- 路由与页面
- 状态上下文
- API 服务层
- 构建链路

这很重要，因为后面你会看到 `Agent` 页面已经不是一个轻量组件，而是承载复杂状态与交互的主工作台。

### 4.4 运行数据层为什么要单独看

像 `data/`、`projects/` 这种目录，不是“杂物堆”，而是理解系统行为的重要入口：

- `data/` 通常包含数据库、论文文件、简报输出、运行时注册信息。
- `projects/` 存放项目工作区，本地执行和研究流程会在这里留下实际痕迹。

如果你以后排查“为什么某个 run 没执行完”“某个工作区写到哪了”，这两个目录非常关键。

## 5. 参考代码对照

### 5.1 `reference/claw-code-main` 的目录启示

`reference/claw-code-main` 在根目录下明确拆出了 `src/`、`rust/`、`tests/` 等层次，说明它把前端界面层和本地原生层作为两个正式边界。对 `ResearchOS` 的启发是：当工作台和本地能力都重要时，目录边界必须能直接反映架构层次。

### 5.2 回到当前仓库

当前仓库更偏 `apps + packages + frontend` 的 Web/API 中心结构，但又保留了 `apps/desktop/` 和工作区执行链路。这种结构说明项目正处在“研究应用”向“研究平台”过渡的阶段。

## 6. 代码精读顺序

1. 根目录 `Get-ChildItem -Force`
2. `apps/` 下的入口文件列表
3. `packages/` 下的一级子目录
4. `frontend/src/` 下的页面、组件、contexts、services
5. `reference/claw-code-main/` 根目录
6. `reference/claw-code-main/src/` 与 `reference/claw-code-main/rust/`

## 7. 动手任务

1. 画出仓库顶层目录图。
2. 给每个顶层目录写一句职责描述。
3. 标出你认为“最可能出运行结果”的目录和“最可能出业务逻辑”的目录。
4. 解释为什么 `reference/` 和真正参与运行的源码目录不能混为一类。

## 8. 验收标准

- 你能解释顶层至少 10 个目录的职责。
- 你能说清楚 `apps/`、`packages/`、`frontend/` 的边界。
- 你能判断一个新文件应该更合理地落在哪一层。

## 9. 常见误区

- 误区一：看到 `packages/` 很大，就把它当成“杂项目录”。
- 误区二：忽视 `data/` 和 `projects/`，导致对运行结果没有直觉。
- 误区三：把 `reference/` 当成本项目代码的一部分来读。
