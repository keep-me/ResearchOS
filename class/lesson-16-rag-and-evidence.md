# 第 16 课：RAG 与证据链

## 1. 本课定位

这一课开始回答一个很多用户最直观的问题：为什么系统能“基于论文库回答问题”。真正有价值的 RAG 不只是检索几段文本，而是构建可信的证据链。

## 2. 学完你应该能回答的问题

- `RAGService` 在项目中的职责是什么。
- 为什么证据链比单纯“回答得像”更重要。
- 论文证据准备和问答生成之间如何衔接。
- 为什么 RAG 是论文库系统和普通聊天系统的重要区别。

## 3. 学习前准备

- 阅读 `packages/ai/research/rag_service.py`。
- 浏览 `packages/ai/paper/paper_evidence.py`。
- 回看论文处理流水线里与证据相关的部分。

## 4. 详细讲解

### 4.1 RAG 在这里不是“外挂搜索”，而是研究问答基础设施

在通用 AI 产品里，RAG 常常只是补知识。但在 `ResearchOS` 里，RAG 更像是论文库的核心交互方式：

- 用户不是在问公开互联网常识。
- 用户通常希望跨论文综合分析。
- 用户希望能追溯结论来源。

所以这里的 RAG 更强调研究资料整合，而不是开放域问答。

### 4.2 为什么证据链如此重要

学术场景和普通聊天最大的不同在于：你不能只看答案通不通顺，还要看它是否有依据。证据链的重要性体现在：

- 方便用户回到原论文验证
- 减少模型随意编造
- 支持跨论文比较
- 为后续写作和报告提供可信引用

这就是为什么 `paper_evidence.py` 这样的模块必须存在。

### 4.3 证据准备是前置资产，不是问答时临时拼接

优秀的论文系统不会把所有工作都留到“用户提问那一刻”才做。前面的论文处理阶段已经在为 RAG 做准备：

- PDF 被解析
- 内容被切分或整理
- 证据片段可被加载
- 相关分析报告可被复用

这样 RAG 才不会每次都从零开始。

### 4.4 RAGService 体现了“检索 + 组织 + 回答”三段式

你在读 `RAGService` 时，应该注意这三层思路：

1. 找到相关论文或相关片段
2. 把片段组织成适合模型理解的上下文
3. 生成回答并保留证据意识

以后你看图谱、Wiki 甚至 workflow 提示词，也会反复看到这种“先组织上下文，再生成”的模式。

### 4.5 为什么这条链路能支撑更多高级能力

一旦证据链做得稳，很多上层能力都更可靠：

- PDF 阅读器中的片段分析
- 多论文综合问答
- 研究综述生成
- 写作辅助中的引用支撑

所以 RAG 不是孤立功能，而是知识生产能力的中心连接点。

## 5. 参考代码对照

### 5.1 对照 `reference/claw-code-main`

`reference/claw-code-main` 更强调研究资产在工作台中的长期使用。对照它可以帮助你理解：RAG 的真正价值不是一次检索，而是把论文内容变成后续可持续调用的知识层。

### 5.2 回到当前仓库

当前仓库的 `packages/ai/paper/` 和 `packages/ai/research/` 已经把“证据准备 -> 上下文组织 -> 回答输出”串成一条完整链路，这才是你读 RAG 时应该盯住的主线。

## 8. 验收标准

- 你能解释 `RAGService` 的高层职责。
- 你能说明证据链的必要性。
- 你能把问答与论文处理阶段联系起来理解。

## 9. 常见误区

- 误区一：把 RAG 理解成“多贴几段文本到 prompt 里”。
- 误区二：只看回答内容，不看证据质量。
- 误区三：忽视前置知识准备对问答质量的影响。
