# 第 07 课：中间件与横切能力

## 1. 本课定位

上一课你知道了入口文件里装了哪些东西，这一课要理解“为什么这些能力要以中间件或公共处理形式存在”。这关系到你未来判断一个功能应该放哪里。

- 什么是横切能力。
- 为什么日志、认证、异常处理、压缩、跨域不应该散在业务函数里。
- `ResearchOS` 里哪些能力已经是统一处理的，哪些还主要留在业务层。

- 重读 `apps/api/main.py` 中的中间件和异常处理部分。

## 4. 详细讲解

### 4.1 什么叫横切能力

横切能力指的是：它不是某一个业务专属，而是会影响很多接口甚至全部接口。例如：

- 记录请求日志
- 识别用户是否已登录
- 统一返回错误格式
- 控制跨域访问
- 对大响应进行压缩

这些事情如果每个 router 都手写一遍，系统很快会失控。

### 4.2 请求日志为什么不能写在每个接口里

如果你把日志写在每个接口函数中，会遇到几个问题：

- 格式不统一
- 有些接口忘了写
- 出错路径不完整
- 无法稳定拿到耗时和统一 request id

`RequestLogMiddleware` 通过包裹整个请求过程，天然更适合做这些事情。这就是“横切能力适合上提”的典型例子。

### 4.3 认证中间件的设计重点

认证中间件的关键不是“判断对不对”，而是“判断策略集中化”。`ResearchOS` 把下面这些决策都集中在入口层：

- 是否启用认证
- 哪些路径白名单放行
- 文档与开放路径如何处理
- 从哪里拿 token
- token 无效时如何返回
- 认证成功后如何把用户信息交给后续请求

你要特别注意：这其实是一种“策略层”设计。业务接口本身不再关心 token 从哪来，而只关心请求已经被过滤过。

### 4.4 统一业务异常的意义

`AppError` 的存在说明项目想把“业务失败”和“框架异常”区分开。统一异常处理有几个实际价值：

- 前端收到更稳定的错误格式
- 日志更容易归类
- 不同模块可以共享错误表达方式
- 接口层不必每次都手写重复 try/except

这也是未来你自己设计服务层时需要学习的习惯。

### 4.5 GZip 与 CORS 解决的是运行环境问题

这两个能力很多人会忽视，因为看起来“不是功能”。但它们实际上决定了系统能不能稳定跑在真实环境中：

- GZip 解决响应体较大时的传输成本。
- CORS 解决前后端地址不同带来的浏览器安全限制。

所以别把它们理解成“可有可无的小优化”，它们是产品可用性的一部分。

## 9. 常见误区

- 误区一：把中间件当成“框架语法”，不思考职责边界。
- 误区二：以为认证只和登录页面有关。
- 误区三：忽视错误格式一致性对前后端协作的价值。
